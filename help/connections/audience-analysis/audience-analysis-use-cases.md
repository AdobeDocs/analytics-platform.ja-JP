# Audience Analysis の使用例 {#analyze-audiences-use-cases}

Audience Analysis は、Customer Journey AnalyticsにおけるExperience Platform オーディエンスメンバーシップデータのレポートを有効にします。 これを実現するには、Audience Analysis 設定ウィザードで管理される設定を使用します。このウィザードは、他のパラメーターや設定の詳細と共に、取り込むプロファイルデータセットを決定するのに役立ちます。 （詳しい概要情報については、[Audience Analysis の概要 &#x200B;](/help/connections/audience-analysis/audience-analysis-overview.md) を参照してください。）

このドキュメントには、Audience Analysis が提供する値をハイライトするユースケースの例が含まれています。 ユースケースを確認する前に、まず、以下のレポートに関する考慮事項に慣れてください。 ユースケースを検討する際は、これらの考慮事項に留意することが重要です。これらの考慮事項は、レポートの最終的な出力に影響を与える可能性があるからです。

## レポートに関する考慮事項

Audience Analysis の初回リリースでは、Experience Platform オーディエンスを処理してCustomer Journey Analyticsに取り込むために必要な基本的な基本作業が確立されています。 分析を実施する際は、Workspace プロジェクト全体の結果に影響を与える可能性のあるいくつかの要因に注意することが重要です。

* **オーディエンスメンバーシップデータは、前日（「昨日」）のみ正確です**：オーディエンスメンバーシップデータには、統合プロファイルサービスによって生成された最新のプロファイルスナップショットデータセットが常に含まれます。 このプロファイルデータセットは毎日のスナップショットで、前日（「昨日」）にのみ正確で、毎晩自動的に再生成および再処理されます。 オーディエンスディメンションは、履歴オーディエンスの状態の再構築ではなく、レポートおよび分類に使用できます。

   * 例：選択したレポート時間枠に関係なく、CJAのレポート可能なオーディエンスは、最新の取り込まれたプロファイルスナップショット（「昨日」）に存在するオーディエンスメンバーシップの状態を常に考慮します。

      * 例えば、レポート期間を「過去 30 日間」に拡大すると、より多くのイベントが含まれ、オーディエンスサイズが変化しているという印象を与えます。 ただし、オーディエンスのプロファイル構成は、選択した期間にかかわらず、常に「昨日」のスナップショットと一致します。

* **ディメンションを含めるには、対応するイベントがディメンションに含まれている必要があります**:Audience Analysis ディメンションを分析できるのは、対応するイベントがCJAに存在する場合のみです。 ビヘイビアー、チャネル、またはライフサイクルモーメントがCJA連携でイベントとして表示されない場合は、分析できません。

   * 例：広告でユーザーをターゲットするために使用されるオーディエンスでは、CJA オーディエンスよりもRTCDP オーディエンスの方が大幅に多くのユーザーが含まれます。 これは、CJA オーディエンスが、レポート期間中にCJAでイベントを受けた人に制限されるからです。

* **ID 解決は、単一の名前空間のみに基づいています**:ID 解決は、Audience Analysis 設定の一部として選択された ID 名前空間に完全に依存します。 分析は、その ID 名前空間に制限され、その名前空間の範囲外のイベントはオーディエンス分析レポートで使用できません。

   * 例：CRM と ECID を組み合わせ、オーディエンス分析設定で CRM ID を使用するステッチされたイベントデータセットの場合、CRM ID を含む行のみがCJAのレポート可能なオーディエンスの一部として認識されます。 したがって、結果のオーディエンスサイズは予想よりも小さくなる可能性があります。

## 例となるユースケース

### ユースケース 1

特定のオーディエンスが特定のチャネル（web やアプリなど）でどのように動作するかを把握し、次のような質問に回答します。

* _「価値の高い見込み客のメンバーは、現在、サイト（ページ、機能、ファネル）で何をしていますか？」_

* _「他のユーザーと比較して、このオーディエンスのキャンペーンとコンテンツのインデックス過剰はどのキャンペーンですか？」_

#### 設定フロー

1. CJAで、1 つの ID 名前空間（ECID や CRM_ID など）と web に焦点を当てたデータビューに対して Audience Analysis を設定します。

   * これにより、毎日のプロファイルスナップショット書き出しを使用して、選択した接続にオーディエンスメンバーシップデータが自動的に取り込まれます

   * イベントデータセットで最もカバレッジが高いと思われる ID 名前空間を選択することをお勧めします

1. Workspace プロジェクトを構築して次の操作を行います。

   * ページ、製品、キャンペーン、デバイスなどでオーディエンス名を分類

   * セッション、コンバージョン率、1 人あたりの売上高などの指標について、オーディエンスと非オーディエンス（または別のオーディエンス）を比較します。

1. チャネル最適化戦略（ターゲティングルール、コンテンツ、オファーのチューニングなど）に関するインサイトをフィードします。

#### ID 解決に関する考慮事項

| ユースケース | コアビジネスの質問 | ID 解決の考慮事項 | 高認証/単一名前空間組織（1 人のユーザー ID の下に既に存在するイベント（例：ログイン /CRM） | フラグメント/マルチ名前空間組織（ECID の下のイベント + CRM およびその他） |
|---------|----------|---------|---------|---------|
| 現在のオーディエンスの行動の詳細 | _「オーディエンス X は現在、チャネル Y で何をしていますか？」 （ページ、ファネル、コンテンツ、オファー）_ | イベントとプロファイルは、最適なカバレッジを実現するために、CJA接続および Audience Analysis 設定で 1 つの一貫性のある ID 名前空間を共有する必要があります。 | ほとんどのアクティビティは 1 つのログイン/CRM ID に既に関連付けられているので、AEPのオーディエンスはCJAの行動データに明確に結合されます。 <p>予測されるレポートギャップを最小限に抑えながら、特定のチャネルで各オーディエンスが何を実行しているかがはっきりとわかります。</p> | ステッチされたデータセットと結合された場合でも、レポートは、設定で選択された単一の ID 名前空間に引き続き制限されます。 <p>一部の顧客が他の ID の下に存在する場合、その行動は含まれず、レポート中に部分的な表示になる可能性があります。</p> |

### ユースケース 2

マーケターやジャーニーデザイナーがオーディエンスの重複を理解することで、エクスペリエンスの重複を排除し、キャンペーン間でのオファーの競合を回避できます。

* _「現在、ロイヤルティメンバー、買い物かごの放棄、チャーンに至る高い傾向の間で重複はどれくらいありますか？」_

* _「今週、価値の高いプロモーションオファーに衝突する可能性が最も高いオーディエンスはどれですか？_

#### 設定フロー

1. RTCDP/AJOのアクティベーションに合わせた単一の ID 名前空間に対して、CJAで Audience Analysis を設定します（例えば、ジャーニーが人物ベースの場合は CRM_ID）。

   * これにより、毎日のプロファイルスナップショット書き出しを使用して、選択した接続にオーディエンスメンバーシップデータが自動的に取り込まれます。

   * これを使用すると、主要なエンゲージメントとコンバージョンレポートを既に強化している既存のデータビューを強化できます。

1. 標準のオーディエンス分析の概要テンプレートと重複のビジュアライゼーションを使用する：

   * オーディエンスの交差やインデックス過剰/インデックス不足の表示を実行します（例：同じロイヤルティゴールドに参加している買い物かごの放棄の割合）。

   * コアディメンション（デバイスタイプ、製品の関心など）でスライスを重ね合わせて、競合が最も重要な場所を把握します。

1. インサイトを使用して、次のような重要な側面を微調整します。

   * RTCDPとAJOのオファーの競合ルールまたはマーケティングアクションルール。

   * オーディエンスの絞り込み（重複が多すぎるターゲット定義を絞り込むなど）。

#### ID 解決に関する考慮事項

| ユースケース | コアビジネスの質問 | ID 解決の考慮事項 | 高認証/単一名前空間組織（1 人のユーザー ID の下に既に存在するイベント（例：ログイン /CRM） | フラグメント/マルチ名前空間組織（ECID の下のイベント + CRM およびその他） |
|---------|----------|---------|---------|---------|
| オーディエンスの重複および衝突検出 | _「オファーの競合を回避するために、現在どのオーディエンスが重複しているか？」_ | 重複は、同じユーザー ID を使用し、CJA接続でアクティビティを使用するオーディエンスに対してのみ計算されます。 | ほとんどのアクティビティは既に 1 つのログイン/CRM ID に関連付けられているので、これにより、オーディエンス間で信頼性の高い重複マップが提供されることが期待されます。 <p>重複グラフは、どのオーディエンスが衝突し、どこに抑制や優先度ルールを適用するかを示す信頼性の高い画像を提供します。</p> | ジャーニーの一部が他の ID の下に存在する場合（例：匿名の ECID のみのブラウジング、コールセンター ID など）、それらのイベントは重複分析に表示されません。 <p>人物が複数の名前空間にまたがって存在する可能性もあります。</p><p>重複は、設定に含まれる ID 名前空間に基づきます。 一部のプロファイルが引き続き ID 間で分割されている場合、重複が真の衝突を過小報告する可能性があります。</p> |

### ユースケース 3

最近主要なオーディエンスを離れた顧客の行動と、その出口に関して行った操作を理解して、次のような質問に回答します。

* _「主なオーディエンスを残したのは誰で、退場する際に何をしたのですか？」_

* _「退社直前に何が起こったのですか。 （エラー、低エンゲージメント、価格変更）」_

#### 設定フロー

1. 1 つの ID 名前空間（例：CRM_ID またはログイン ID）と関連するデータビュー（web、アプリ、CRM など）に対してCJAで Audience Analysis を設定します。

   * これにより、毎日のプロファイルスナップショット書き出しを使用して、選択した接続にオーディエンスメンバーシップデータが自動的に取り込まれます。

   * イベントデータセットで最もカバレッジが高いと思われる ID 名前空間を選択することをお勧めします。

1. オーディエンス分析/オーディエンスの概要テンプレート（_昨日_ に固定）で、次を使用します。

   * 現在のメンバー：オーディエンスにまだ残っているメンバー

   * 退出したオーディエンス：昨日退出したオーディエンス

1. Workspace プロジェクトを構築して次の操作を行います。

   * 昨日オーディエンス X を終了したプロファイルをフィルタリングした後、次を確認します。

      * 出口に至るまでの行動（前回のセッション、エラー、価格/オファーの露出、チャネルミックス）。

      * 彼らの出口後の行動（彼らは製品を切り替え、ダウングレード、休眠しました）。

   * 離脱したコホートを地域、デバイス、テニュア、バリュー階層ごとに分類して、影響の大きいポケットを見つけます。

1. RTCDPまたはAJOで、ジャーニーの更新とオーディエンスの獲得設定にインサイトを使用します。

#### ID 解決に関する考慮事項

| ユースケース | コアビジネスの質問 | ID 解決の考慮事項 | 高認証/単一名前空間組織（1 人のユーザー ID の下に既に存在するイベント（例：ログイン /CRM） | フラグメント/マルチ名前空間組織（ECID の下のイベント + CRM およびその他） |
|---------|----------|---------|---------|---------|
| 退出したオーディエンス – チャーン分析 | _「誰が主要なオーディエンスを残しましたか？」_ <p>_「彼らは出口を出る際に何をしましたか？」_</p> | オーディエンスの離脱は、接続およびオーディエンス設定に使用されたのと同じユーザー ID でトラッキングされます。 | 安定したログイン/CRM ID で測定された EXIT は、真の行動変化を反映する傾向があります。 <p>この ID を使用してオーディエンスを離れる場合、通常は実際の変化（チャーン、ダウングレード、非アクティブ）を意味します。</p><p>ジャーニーの最近の動作を分析して、ジャーニーを微調整し、自信を持ってオファーを獲得できます。</p> | 出口は、プロファイルとイベントが設定済み ID を共有する場合にのみ表示されるので、慎重に解釈する必要があります。<p>離脱済みコホートを強力なヒントまたはシグナルとして使用しますが、重要な決定を行う前に他のデータポイントとクロスチェックすることをお勧めします。</p> |
